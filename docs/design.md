# JdkManagement 设计说明

## 1. 产品定位

JdkManagement 是一个面向 Windows 的多运行时管理工具，目标不是只管理 Java，而是统一管理常见开发运行时与工具链。

当前聚焦范围：

- Java
- Python
- Node.js
- Go
- Maven
- Gradle

工具形态以 CLI 为主，强调可追踪的安装过程、清晰的状态模型，以及项目级覆盖能力。

## 2. 设计目标

核心目标：

- 提供统一的安装、卸载、查询、切换命令面
- 支持用户级默认选择和项目级局部覆盖
- 让下载、校验、解压、激活等阶段具备一致的日志风格
- 保持缓存、配置、锁定文件等状态可检查、可清理、可回放

非目标：

- 当前阶段不追求跨平台一致实现
- 当前阶段不引入复杂的后台服务或守护进程
- 当前阶段不在文档中扩展未实现的行为

## 3. 状态模型

用户级状态位于数据根目录下的 `state`，项目级状态位于工作目录下的 `.jkm`。

关键文件：

- `state/active_runtimes.tsv`
- `state/settings.tsv`
- `.jkm/local_runtimes.tsv`
- `.jkm/project.lock.tsv`

设计原则：

- 用户级选择代表默认运行时
- 项目级选择只覆盖当前项目树
- 锁定文件用于重建项目所需的运行时组合

## 4. 配置模型

配置统一通过 `jkm config` 管理，配置文件使用简单的键值存储。

当前配置分为两类：

- 网络配置：代理、证书
- 源地址配置：不同 provider 的镜像基础地址

这样做的目的：

- 所有 provider 共享同一套下载入口
- 镜像切换不需要修改 provider 代码
- 代理和证书行为在不同运行时之间保持一致

## 5. 下载子系统

下载逻辑集中在共享 helper 中，由所有 provider 复用。

当前能力：

- 分阶段下载日志
- 平滑速率显示
- 早期 ETA 门控
- 自动重试
- `.partial` 续传
- 完成后再落到最终目标路径

设计原则：

- 展示层和下载层在同一 helper 中收口
- provider 只负责版本解析和 URL 生成
- 通用网络行为不在单个 provider 中重复实现

## 6. Provider 边界

每个 provider 负责：

- 远程版本枚举
- 版本选择和下载地址计算
- 解压和安装布局
- 必要的校验信息获取

共享层负责：

- 下载
- 网络环境初始化
- 日志输出格式
- 速率和 ETA 平滑
- 断点续传与重试

## 7. Python 模型

Python 采用“基础解释器 + 命名环境”的结构。

状态上区分：

- 基础解释器
- `base`
- 具名环境

这样设计的原因：

- 能与其他运行时共用统一的 CLI 结构
- 允许项目锁定文件同时表达基础版本和环境名称
- 便于 `sync` 先安装基础解释器，再补齐环境

## 8. 缓存策略

缓存分为两个区域：

- `cache/downloads`
- `cache/temp`

配套命令：

- `jkm cache list`
- `jkm cache clear`
- `jkm cache prune`

设计原则：

- 用户需要能看到缓存占用
- 用户需要能整体清空，也能按策略裁剪
- 清理逻辑尽量不影响已安装运行时

## 9. 项目锁定与同步

项目锁定文件记录项目当前使用的托管运行时组合。

当前流程：

1. 开发者在项目目录选择本地运行时
2. 执行 `jkm lock write` 生成 `.jkm/project.lock.tsv`
3. 其他机器执行 `jkm sync`
4. `sync` 安装缺失运行时，并写回本地或用户级选择

特殊处理：

- 保留的外部运行时不写入锁定文件
- Python 环境条目需要先确保基础解释器存在
